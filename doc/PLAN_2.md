## Критические задачи для исправления (P2-P3)

### 1. Orphan 3x-ui clients (P2 - Reliability) ⚠️
- **Файл:** `internal/bot/subscription.go:26`
- **Проблема:** При пересоздании подписки старый клиент в 3x-ui не удаляется
- **Решение:** Перед созданием нового клиента вызвать `xui.DeleteClient()` для старого
- **Влияние:** Предотвращает накопление orphan clients и исчерпание лимитов в 3x-ui

### 2. GetAllTelegramIDs без пагинации (P3 - Memory) ⚠️
- **Файл:** `internal/bot/admin.go:183`
- **Проблема:** При broadcast все Telegram ID загружаются в память
- **Решение:** Batch processing с cursor-based pagination или streaming
- **Влияние:** Снижение памяти для больших баз пользователей (10k+)

### 3. Retry jitter (P4 - Code Quality) ⚠️
- **Файл:** `internal/xui/client.go:455-482`
- **Проблема:** Exponential backoff без jitter приводит к thundering herd
- **Решение:** Добавить `time.Duration(rand.Intn(1000)) * time.Millisecond` к задержке
- **Влияние:** Предотвращает одновременные повторные запросы при массовых сбоях

## Долгосрочная ключевая функция (3-6 месяцев)

### 11. Мульти-серверность 🌐
**Концепция:** Один клиент на ВСЕХ серверах, ОДНА подписка со ВСЕМИ серверами

**Как это работает:**
1. Бот генерирует UUID, email, subId ОДИН РАЗ
2. Для каждого сервера создаёт клиента с ТЕМИ ЖЕ данными
3. Бот генерирует VLESS конфиг со всеми серверами
4. Пользователь получает ОДНУ подписку для всех клиентов

**HTTP Endpoint:**
- Бот запускает HTTP сервер на порту 8080
- Endpoint: `GET /sub/{subID}`
- Возвращает base64-закодированный конфиг xray/v2ray
