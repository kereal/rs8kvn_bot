## 🎯 Цель

Реализовать HTTP endpoint, который:

* принимает запрос от клиента с кодом подписки
* валидирует этот код по внутренней базе
* получает оригинальную подписку из 3xUI
* модифицирует подписку (тело и заголовки)
* возвращает клиенту итоговый ответ
* использует кэш для повышения производительности и надежности

---

## 🧩 Точка входа

HTTP GET запрос:

/sub/{subscription_code}

Где:

* `subscription_code` — уникальный идентификатор подписки

---

## 🧩 Валидация запроса

После получения запроса необходимо:

1. Проверить наличие `subscription_code`
2. Проверить существование записи в базе (таблица subscriptions)
3. Проверить корректность и статус подписки (если применимо)

Если проверка не пройдена:
→ вернуть корректный HTTP-ответ с ошибкой (например, 404 или 403)

---

## 🧩 Основной процесс

### 🔀 Flow (diagram)

```mermaid
flowchart TD
    A[Client] --> B[/sub/{subscription_code}]
    B --> C[Validate in DB]
    C --> D{Cache valid?}
    D -- Yes --> E[Return cached body + headers]
    D -- No --> F[Fetch from 3xUI]
    F --> G[Get body + headers]
    G --> H[Detect format]
    H --> I[Modify body + headers]
    I --> J[Save to cache]
    J --> K[Return response]

    F -. error .-> L{Cache exists?}
    L -- Yes --> E
    L -- No --> M[Return error]
```

1. Принять HTTP-запрос от клиента

2. Валидировать `subscription_code` через базу

3. Проверить кэш (см. раздел кэширования)

4. Если кэш валиден:
   → вернуть кэшированный ответ (тело + заголовки)

5. Если кэш отсутствует или истёк:

   * выполнить HTTP GET в 3xUI
   * получить:

     * тело ответа
     * все HTTP-заголовки

6. Определить формат подписки:

   * base64
   * plain text

7. Модифицировать подписку:

   ### Тело:

   * добавить произвольное количество серверов из конфигурации
   * объединить с оригинальной подпиской
   * сохранить исходный формат

   ### Заголовки:

   * использовать оригинальные заголовки как основу
   * при необходимости дополнить или изменить

8. Сформировать итоговый ответ

9. Сохранить результат в кэш

10. Вернуть клиенту:

* итоговое тело
* итоговые HTTP-заголовки

---

## ⚠️ Требования к HTTP-заголовкам

1. Все заголовки ответа от 3xUI должны быть получены и учтены
2. По умолчанию заголовки должны быть сохранены без изменений
3. Допускается модификация и расширение заголовков
4. Нельзя удалять критически важные заголовки

К критически важным относятся:

* Content-Type
* Content-Disposition
* Subscription-Userinfo
* Profile-Update-Interval
* любые служебные заголовки

5. Должна корректно обрабатываться длина ответа и служебные параметры

---

## ⚙️ Конфигурация

Система должна поддерживать:

* флаг включения/выключения функциональности
* список добавляемых серверов (любое количество)
* порядок добавления (в начало или в конец)
* параметры кэширования

---

## 🧠 Требования к модификации подписки

### Тело:

1. Оригинальные данные не должны изменяться
2. Добавление серверов не должно нарушать формат
3. Формат ответа должен совпадать с исходным
4. Не допускается добавление некорректных или пустых записей

### Заголовки:

1. Оригинальные заголовки должны быть базой
2. Модификации должны быть контролируемыми
3. Нельзя нарушать совместимость с клиентами

---

## 🗄️ Кэширование (упрощённое)

### Цель:

* снизить нагрузку на 3xUI
* ускорить ответы
* повысить устойчивость

---

### Общие требования:

1. Кэшируется итоговый ответ:

   * тело
   * HTTP-заголовки

2. Ключ кэша:

   * `subscription_code`

---

### Поведение:

#### 1. Есть валидный кэш (TTL не истёк)

→ вернуть кэшированный ответ

#### 2. Кэш отсутствует или истёк

→ выполнить полный запрос в 3xUI
→ обработать
→ сохранить в кэш
→ вернуть клиенту

---

### TTL:

* должен быть настраиваемым
* рекомендуемое значение: 60–120 секунд

---

### Поведение при ошибках:

1. Ошибка при запросе к 3xUI:

   * если есть кэш → вернуть кэш
   * если нет → вернуть ошибку

2. Ошибка обработки:

   * если есть кэш → вернуть кэш
   * иначе → вернуть оригинальный ответ (если получен)

---

### Дополнительные требования:

* кэш должен хранить тело и заголовки вместе
* кэш должен быть консистентным (обновляется целиком)
* кэш должен быть прозрачным для клиента

---

## 🛡️ Надежность

При любой ошибке:

* если получен ответ от 3xUI:
  → вернуть оригинальный ответ без изменений

* если ответа нет:
  → вернуть корректную HTTP-ошибку

---

## 🧪 Требования к тестированию

1. Валидный `subscription_code`
2. Невалидный код
3. Base64 подписка
4. Plain text подписка
5. Пустая подписка
6. Некорректные данные
7. Отключенный функционал
8. Модификация тела
9. Модификация заголовков
10. Работа кэша (hit/miss)
11. Истечение TTL
12. Ошибка 3xUI + наличие кэша
13. Ошибка 3xUI + отсутствие кэша
14. Полное сохранение критических заголовков
