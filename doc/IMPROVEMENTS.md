- Установите и запустите приложение

**2. Импорт подписки**

- Выберите «Вставить из буфера»

- Дождитесь загрузки конфигурации

**Автоматическая маршрутизация:**
- Зарубежные сайты работают через VPN
- Российские сервисы — напрямую
- VPN можно не выключать

**Предупреждение:**
- Если используете другое приложение — не заходите через VPN на российские ресурсы
- Это может привести к блокировке сервера

### 4. Мульти-серверность 🌐 [ВЫСОКИЙ ПРИОРИТЕТ] 🆕

> **КЛЮЧЕВАЯ ФИЧА:** Один клиент на ВСЕХ серверах, ОДНА подписка со ВСЕМИ серверами в Happ

---

#### 4.1 Как это работает с 3x-ui

**Контекст:**
- Каждый сервер = своя 3x-ui панель (панели НЕ связаны между собой)
- Нужно создать одного клиента на ВСЕХ серверах
- Клиент получает ОДНУ подписку со ВСЕМИ серверами

**Ключевое решение — одинаковые данные на всех серверах:**
- **Один UUID** — генерируем один раз, используем везде
- **Один email** — например, `tg_123456789` (telegram ID)
- **Один subId** — идентификатор подписки
- **Одинаковые лимиты** — трафик и срок действия

**Процесс создания подписки:**
1. Пользователь нажимает "Получить подписку"
2. Генерируем UUID, email, subId (ОДИН РАЗ)
3. Перебираем все серверы из конфига
4. Для каждого сервера создаём клиента с ТЕМИ ЖЕ UUID, email, subId
5. Генерируем объединённую подписку для Happ
6. Сохраняем в БД

**Что нужно от каждого сервера:**
- Адрес панели (host)
- Логин/пароль админа 3x-ui
- ID inbound (обычно 1)
- Reality настройки: public_key, short_id, server_name (SNI)
- Эти настройки берутся из 3x-ui панели → Inbound → Settings

---

#### 4.2 Конфигурация серверов

**Решение:** Хранить серверы в конфиге (YAML), НЕ в базе данных.

**Почему в конфиге:**
- Проще — нет админ-команд в боте
- Безопаснее — пароли не в БД
- Статично — серверы редко меняются
- Легко деплоить — один файл конфигурации

**Структура конфига (config.yaml):**
- name — название сервера ("EU-Netherlands", "US-NewYork")
- host — адрес панели ("https://server.com:2053")
- username/password — доступ к 3x-ui
- inbound_id — ID inbound в 3x-ui
- public_key — Reality публичный ключ
- short_id — Reality short ID
- server_name — SNI (например, "www.google.com")
- server_port — порт (обычно 443)

**Переменные окружения:**
- Пароли хранить в ENV: `SERVER_EU_PASSWORD`, `SERVER_US_PASSWORD`
- В конфиге использовать: `password: "${SERVER_EU_PASSWORD}"`

---

#### 4.3 Генерация объединённой подписки

**Проблема:** 3x-ui отдаёт конфиг для ОДНОГО сервера. Нам нужен конфиг для ВСЕХ серверов.

**Решение:** Бот сам генерирует VLESS конфиг со всеми серверами.

**HTTP Endpoint в боте:**
- Бот запускает HTTP сервер на отдельном порту (например, 8080)
- Endpoint: `GET /sub/{subID}`
- Возвращает base64-закодированный конфиг xray/v2ray
- **Авторизация:** Не требуется — subID это уникальный UUID (достаточно секретный)

**Формат конфига:**
- **outbounds** — список всех серверов как VLESS outbounds
  - Каждый outbound = один сервер
  - Все используют один UUID
  - Reality настройки для каждого сервера свои
- **routing** — правила маршрутизации
  - Прямой доступ к российским сайтам (geosite:category-ru)
  - Прямой доступ к локальным адресам (geoip:private)
  - Остальное через основной сервер
- **direct** — outbound для прямого подключения

**Как клиент использует конфиг:**
- Импортирует подписку в Happ
- Видит все серверы в списке
- Может переключаться между ними
- Автоматический fallback при недоступности

---

#### 4.4 Обработка ошибок

**Если сервер недоступен при создании подписки:**
- Продолжить создание на других серверах
- Записать в лог предупреждение
- НЕ показывать ошибку пользователю (подписка всё равно работает)
- Уведомить админа через бот

**Если все сервера недоступны:**
- Показать пользователю: "Сервис временно недоступен, попробуйте позже"
- Не создавать запись в БД
- Критическое уведомление админу

**При удалении подписки:**
- Удалить клиента со ВСЕХ серверов
- Если сервер недоступен — игнорировать, пометить в лог

---

#### 4.5 Преимущества подхода

✅ **Пользователь получает ОДНУ подписку**
✅ **В Happ видны ВСЕ серверы**
✅ **Можно переключаться между серверами**
✅ **Автоматический fallback при падении сервера**
✅ **UUID одинаковый — проще управлять**
✅ **Не нужны админ-команды в боте**
✅ **Бот остаётся лёгким и быстрым**
✅ **Конфигурация статичная — меньше ошибок**

#### 4.6 Ограничения

⚠️ **Нужны Reality настройки для КАЖДОГО сервера**
- Берутся из 3x-ui панели: Inbound → Settings → Reality
- public_key, short_id, server_name

⚠️ **Трафик считается на каждом сервере отдельно**
- Нельзя синхронизировать трафик между серверами

- Или использует основной (первый в списке)

⚠️ **Добавление сервера требует рестарта бота**
- Серверы в конфиге, не в БД
- Но это нормально — серверы редко добавляются

#### 4.10 Миграция существующих пользователей

**Проблема:** Текущие пользователи имеют подписки на одном сервере, нужно перенести их на мульти-серверную архитектуру.

**Решение:**
- При следующем продлении подписки пользователь автоматически переводится на мульти-серверную схему
- Старые подписки продолжают работать до истечения
- Уведомление пользователя о новых серверах при продлении

**Процесс:**
1. Пользователь продлевает подписку
2. Бот проверяет, есть ли подписка на других серверах
3. Если нет — создаёт клиента на всех серверах с ТЕМ ЖЕ UUID
4. Генерирует новую объединённую подписку
5. Отправляет пользователю новую ссылку

**Время миграции:** Зависит от частоты продления подписок (обычно 1 месяц)

---

#### 4.11 Добавление нового сервера

**Сценарий:** Нужно добавить новый сервер в уже работающую систему с существующими пользователями.

**Процесс:**
1. Добавить конфигурацию нового сервера в config.yaml
2. Перезапустить бота
3. Запустить скрипт миграции или дождаться естественной миграции при продлении

**Автоматическая миграция (опционально):**
- Скрипт проходит по всем активным подпискам
- Создаёт клиента на новом сервере с существующим UUID
- Обновляет подписку (пользователь увидит новый сервер при следующем обновлении)

**Важно:** Не пересылать пользователям новые ссылки автоматически — они получат их при следующем продлении

---

#### 4.12 Обновление подписки

**Сценарий:** Пользователь продлевает подписку — нужно обновить лимиты на всех серверах.

**Процесс:**
1. Получить список всех серверов из конфига
2. Для каждого сервера обновить лимиты клиента (traffic + expiration)
3. Если сервер недоступен — retry с exponential backoff (3 попытки)
4. Записать в БД статус обновления каждого сервера
5. Уведомить пользователя об успешном продлении

**Если часть серверов недоступна:**
- Продление считается успешным, если обновлен хотя бы один сервер
- Недоступные серверы обновляются при восстановлении (background job)

---

#### 4.13 Disaster Recovery

**Сценарий:** Сервер вышел из строя или панель 3x-ui недоступна.

**Для новых подписок:**
- Автоматический skip недоступного сервера
- Подписка создаётся на доступных серверах

**Для существующих подписок:**
- Пользователь может переключиться на другой сервер в Happ
- Fallback работает автоматически

**Восстановление сервера:**
- При восстановлении сервера запустить скрипт синхронизации
- Скрипт создаёт всех клиентов на восстановленном сервере
- Альтернатива: клиенты создаются автоматически при следующем продлении

**Мониторинг:**
- Health check для каждого сервера (каждую минуту)
- Алерт админу при недоступности сервера > 5 минут
- Prometheus метрики: server_status, server_response_time
