# Документация: процесс синхронизации подписок с VPN-нодами

## 1. Назначение

Этот документ описывает текущий процесс синхронизации подписок с VPN-нодами в проекте, его реальные инварианты, ограничения, уже исправленные проблемы и известные оставшиеся риски.

Документ намеренно фиксирует не только "как должно быть", но и "как работает сейчас", включая спорные места, которые пока оставлены на будущее.

## 2. Где находится логика

Основные файлы:

- `internal/service/sync.go` — основной orchestration sync-процесса.
- `internal/database/subscription_nodes.go` — хранение состояния синхронизации в БД.
- `internal/scheduler/subscription_sync_worker.go` — фоновый запуск retry.
- `internal/service/subscription.go` — запуск синхронизации из lifecycle подписки.
- `internal/service/order.go` — запуск синхронизации при активации продукта.
- `internal/vpn/client.go` — VPN-интерфейс и классификация типовых ошибок адаптера.
- `internal/vpn/threex_ui.go` — адаптер 3x-ui.
- `internal/vpn/proxman.go` — адаптер proxman.

## 3. Базовая модель

### 3.1. Источник истины

Таблица `subscription_nodes` хранит состояние синхронизации для пары:

- `subscription_id`
- `node_id`

Она используется одновременно как:

- очередь операций;
- текущее состояние синхронизации;
- журнал последней ошибки и retry-метаданных.

Это не идеальная модель, но именно она является текущим источником истины для sync-процесса.

### 3.2. Статусы

Сейчас код использует 4 статуса:

- `active`
- `pending_add`
- `pending_remove`
- `pending_update`

Семантика:

- `active` — на этой ноде для подписки считается успешно примененная актуальная конфигурация.
- `pending_add` — требуется создать доступ на ноде.
- `pending_update` — требуется обновить уже существующий доступ на ноде.
- `pending_remove` — требуется удалить доступ с ноды.

### 3.3. Состояния и переходы

Текущая state machine такая:

- `pending_add -> active` после успешного `CreateSubscription()`.
- `pending_add -> active` после `CreateSubscription()` с ошибкой "already exists", если последующий `UpdateSubscription()` завершился успешно.
- `pending_update -> active` после успешного `UpdateSubscription()`.
- `pending_remove -> delete row` после успешного `DeleteSubscription()`.
- `pending_remove -> delete row` после idempotent-ошибки "not found".
- любой `pending_*` при ошибке остается в том же pending-статусе и получает `retry_count`, `retry_at`, `last_error`.

Terminal-state сейчас нет.

### 3.4. Поддержка нескольких типов нод

Сейчас система поддерживает три типа VPN-нод:

- `3x-ui` — полнофункциональный адаптер с созданием/обновлением/удалением клиентов через HTTP API панели.
- `proxman` — адаптер, отправляющий HTTP webhook-события для create и delete. `UpdateSubscription` — no-op (возвращает `nil`), т.к. webhook-payload не содержит полей traffic/expiry.
- `fetch` — read-only адаптер, все методы (Create/Update/Delete) — no-op. `subscription_url` используется subserver'ом напрямую, без добавления subID. Не управляет VPN-клиентами.

Важно:

- `subscription_nodes` теперь создаётся для всех активных нод плана, независимо от типа;
- при обработке pending-записей система проверяет тип ноды и использует соответствующий обработчик;
- при неизвестном типе ноды пишется предупреждение в лог, вызывается `handleSyncError` (retry_count++, retry_at, last_error), запись остаётся в pending-статусе;
- в `syncNodes()` тип ноды определяется по рантайм-списку `s.nodes`, который заполняется при старте бота.

## 4. Как запускается синхронизация

### 4.1. При создании подписки

Поток:

1. Создается подписка.
2. `ensureSubscriptionNodes()` загружает активные ноды плана.
3. Для отсутствующих пар `subscription x node` создаются строки `pending_add`.
4. Если `syncService != nil`, вызывается best-effort `SyncSubscription()`.

Важно:

- ошибки синхронизации не откатывают создание подписки;
- это asynchronous best-effort модель.

### 4.2. При смене плана / продлении / downgrade

Поток используется из `RenewSubscription()`, `ExpireSubscription()` и `OrderService.ActivateProduct()`:

1. Бизнес-операция меняет подписку.
2. `MarkActiveNodesPendingUpdate()` переводит общие активные ноды нового набора в `pending_update`.
3. `ReconcilePlanNodes()` достраивает нужные `pending_add` и `pending_remove`.
4. `SyncSubscription()` пытается сразу обработать due pending-строки.

Важно:

- изменение плана считается успешным даже если sync на нодах не завершился;
- фактическая консистентность между подпиской и нодами достигается eventual consistency-моделью.

### 4.3. При удалении подписки

Используется `MarkAllForRemoval()`:

- `active`, `pending_add`, `pending_remove` переводятся в `pending_remove`;
- затем вызывается sync, который удаляет клиентов с нод.

### 4.4. Фоновый retry `SyncPendingNodes`

`SubscriptionSyncWorker` периодически обрабатывает due-записи `pending_add`, `pending_update` и `pending_remove`:

1. Загружает due pending-записи из `subscription_nodes`.
2. Группирует их по подписке.
3. Повторно загружает подписку и применяет операции к соответствующим нодам.
4. При успехе переводит записи в `active` либо удаляет `pending_remove`.
5. При ошибке сохраняет `retry_count`, `retry_at` и `last_error`; следующие попытки используют exponential backoff.
6. Отдельно reconciles orphaned clients и строки, для которых подписка больше не существует.

Ошибки отдельных подписок логируются и не останавливают обработку остальных; отмена контекста завершает текущий проход.

### 4.5. Напоминания об истечении подписки

`SubscriptionReminderWorker`:

- делает один проход сразу при старте;
- затем сканирует базу каждые 30 минут;
- выбирает активные платные подписки, у которых `expires_at` попадает в окно 3д / 1д / 3ч до истечения;
- для найденных подписок атомарно claim-ит соответствующий bit с проверкой текущего `expires_at`, затем отправляет напоминание через `SubscriptionService.SendExpiryReminder()`;
- при ошибке Telegram освобождает claim, чтобы следующая попытка могла повториться;
- при продлении reset `reminders_sent` выполняется в одной транзакции с обновлением подписки и созданием order.

Важно:

- в рассмотрение попадают только `status = 'active'`, `expires_at IS NOT NULL` и планы, отличные от `free` и `trial`;
- атомарный claim предотвращает повторы при пересекающихся 30-минутных окнах и параллельных worker-вызовах;
- при продлении подписки поле `reminders_sent` сбрасывается в `0`, поэтому новая подписная история снова проходит весь триплет напоминаний;
- если отправка сообщения пользователю падает, воркер логирует ошибку, освобождает claim и продолжает сканирование;
- `SubscriptionReminderWorker` регистрируется в `cmd/bot/main.go` как отдельный фоновый воркер с `recoverAndReport("Subscription reminder worker")` и учитывается в `wg.Add(...)`.

Метрики:

- `subscription_reminders_total{window,result}` — отправки по окну (`3d`, `1d`, `3h`) и результату (`success`, `error`);
- `subscription_reminder_runs_total` — число проходов worker.

## 5. Как работает reconcile


`ReconcilePlanNodes()` сравнивает:

- целевой набор активных нод плана;
- текущие строки `subscription_nodes`.

Логика:

1. Если нода есть в плане, но отсутствует в `subscription_nodes`:
   - создается `pending_add`.
2. Если нода есть в плане и уже `active`:
   - ничего не делается.
3. Если нода есть в плане и уже `pending_add`:
   - ничего не делается.
4. Если нода есть в плане и уже `pending_update`:
   - состояние сохраняется как есть.
5. Если нода есть в плане и была `pending_remove`:
   - переводится обратно в `pending_add`.
6. Если нода была `active`, но больше не входит в план:
   - переводится в `pending_remove`.
7. Если нода была `pending_add`, но больше не входит в план:
   - обычно переводится в `pending_remove`.
   - исключение: если `targetSet` пуст, `pending_add` сейчас сохраняется как есть.

Последний пункт — известное спорное поведение, оно пока оставлено без изменений.

## 6. Как выполняются операции на нодах

`syncNodes()` сначала определяет тип ноды через рантайм-список:

- если нода отсутствует в `s.nodes` — пропускается с предупреждением;
- если тип `3x-ui`, `proxman` или `fetch` — обрабатывается соответствующим адаптером;
- если тип неизвестен — пишется предупреждение, вызывается `handleSyncError` (retry_count++, retry_at, last_error), запись остаётся pending.

### 6.1. `pending_add`

`processPendingAdd()`:

1. Ищет runtime VPN client для `node_id`.
2. Если runtime client недоступен, вызывается retry-ветка, а запись не удаляется.
3. Загружает план и собирает `vpn.SubscriptionProvision`.
4. Вызывает `CreateSubscription()`.
5. Если create успешен:
   - строка переводится в `active`.
6. Если create вернул `vpn.ErrSubscriptionAlreadyExists`:
   - вызывается `UpdateSubscription()`.
   - только если update успешен, строка переводится в `active`.
   - если update падает, запись остается `pending_add`, записывается retry.
7. Любая другая ошибка:
   - pending-строка сохраняется;
   - записываются retry-метаданные.

### 6.2. `pending_update`

`processPendingUpdate()`:

1. Ищет runtime VPN client.
2. Если client недоступен, запись не удаляется, а уходит в retry.
3. Загружает план и формирует актуальный `SubscriptionProvision`.
4. Вызывает `UpdateSubscription()`.
5. При успехе переводит запись в `active`.
6. При ошибке оставляет `pending_update` и пишет retry-метаданные.

### 6.3. `pending_remove`

`processPendingRemove()`:

1. Ищет runtime VPN client.
2. Если client недоступен, запись не удаляется, а уходит в retry.
3. Вызывает `DeleteSubscription()`.
4. Если адаптер вернул `vpn.ErrSubscriptionNotFound`:
   - это считается идемпотентным успехом;
   - строка удаляется.
5. При успешном delete:
   - строка удаляется.
6. При другой ошибке:
   - запись остается `pending_remove`;
   - пишется retry.

## 7. Retry и ошибки

### 7.1. Retry policy

`CalculateRetryAt()` сейчас использует capped backoff:

- `1m`
- `2m`
- `5m`
- `15m`
- `30m`
- `45m`
- далее всегда `60m`

Поведение:

- после неудачи статус не меняется;
- увеличивается `retry_count`;
- обновляется `retry_at`;
- сохраняется `last_error`.

Важно:

- upper bound по числу retry нет;
- бесконечно failing задача будет бесконечно оставаться pending.

### 7.2. Агрегация ошибок

Раньше sync возвращал только последнюю ошибку.

Сейчас:

- `syncNodes()` собирает ошибки через `errors.Join(...)`;
- `SyncPendingNodes()` тоже агрегирует ошибки через `errors.Join(...)`.

Это улучшило диагностику частичных сбоев по одной подписке.

### 7.3. Идемпотентные ошибки VPN

Раньше `service/sync.go` определял идемпотентность через `strings.Contains(err.Error(), ...)`.

Сейчас это исправлено частично:

- на уровне `vpn` введены sentinel errors:
  - `vpn.ErrSubscriptionAlreadyExists`
  - `vpn.ErrSubscriptionNotFound`
  - `vpn.ErrNotImplemented`
- `service/sync.go` больше не знает про тексты ошибок и использует только `errors.Is(...)`.

Важно:

- текстовая классификация полностью не исчезла, она просто локализована внутри пакета `vpn`;
- это лучше прежнего, потому что хрупкость теперь сосредоточена в одном месте.

## 8. Защита от двойной обработки

Сейчас в `SyncService` есть самый простой per-subscription lock:

- `lockSubscription(subscriptionID)`
- используется и в `SyncSubscription()`, и в `SyncPendingNodes()`.

Что это дает:

- внутри одного процесса две параллельные обработки одной подписки не выполняются одновременно;
- снижается риск двойного `CreateSubscription` / `UpdateSubscription` / `DeleteSubscription`;
- снижается риск гонки между ручным sync и фоновым воркером.

Ограничение:

- lock только in-process;
- если когда-либо будет несколько процессов или реплик, этой защиты будет недостаточно.

## 9. Что уже было исправлено

Ниже перечислены проблемы, которые были найдены в ходе аудита и уже исправлены.

### 9.1. Ложный `active` при `already exists + failed update`

Раньше:

- `pending_add` мог перейти в `active`, даже если fallback `UpdateSubscription()` завершался ошибкой.

Сейчас:

- `active` выставляется только после успешного `CreateSubscription()` или успешного fallback `UpdateSubscription()`.

### 9.2. Silent delete pending-записей при отсутствии runtime client

Раньше:

- если runtime VPN client отсутствовал, а нода была недоступна/невалидна, pending-запись могла быть удалена.

Сейчас:

- вместо удаления запись остается pending и уходит в retry.

### 9.3. Возврат только последней ошибки

Раньше:

- частичные ошибки терялись.

Сейчас:

- ошибки агрегируются через `errors.Join`.

### 9.4. Текстовый matching в `service`

Раньше:

- `service/sync.go` сам анализировал тексты ошибок.

Сейчас:

- `service` использует `errors.Is`, а классификация перенесена в `vpn`.

### 9.5. Неконсистентность схемы для `pending_update`

Для поддержки `pending_update` используется отдельная миграция:

- `internal/database/migrations/027_add_pending_update_sync_status.up.sql`

Также добавлен down-файл:

- `internal/database/migrations/027_add_pending_update_sync_status.down.sql`

Важно:

- уже примененную миграцию `018` редактировать нельзя;
- схема должна обновляться через новую миграцию, а не через переписывание старой.

Для expiry reminders используется отдельная миграция:

- `internal/database/migrations/030_add_reminders_sent_to_subscriptions.up.sql`
- `internal/database/migrations/030_add_reminders_sent_to_subscriptions.down.sql`

Колонка `subscriptions.reminders_sent` хранит bitmask отправленных касаний (`3d`, `1d`, `3h`).

## 10. Известные проблемы, оставленные на будущее

Ниже перечислены проблемы, которые известны, но пока сознательно не исправляются.

### 10.1. Классификация ошибок все еще частично текстовая

Хотя `service` больше не анализирует строки, пакет `vpn` пока все еще использует `strings.Contains(...)` для преобразования внешних ошибок в sentinel errors.

Риск:

- при изменении текстов ошибок в конкретном VPN backend поведение может поменяться.

Почему пока оставлено:

- это уже локализовано в одном месте;
- для текущей архитектуры это минимально достаточный компромисс.

### 10.2. Lock только внутри процесса

Риск:

- при нескольких процессах или репликах возможна двойная обработка одной подписки.

Почему пока оставлено:

- проект сейчас single-binary / single-process;
- in-process mutex — самый простой практический уровень защиты на текущем этапе.

### 10.3. Нет terminal-state для вечно failing задач

Риск:

- broken pending-задача может бесконечно ретраиться раз в 60 минут.

Почему пока оставлено:

- это упрощает модель;
- не требуется вводить дополнительный статус и отдельный remediation-flow.

### 10.4. Спорная логика при пустом `targetSet`

Сейчас `ReconcilePlanNodes()` сохраняет `pending_add`, если целевой набор нод пуст.

Риск:

- могут оставаться висящие pending-записи;
- семантика пустого плана до конца не формализована.

Почему пока оставлено:

- не хочется сломать защитное поведение без отдельного решения по бизнес-семантике пустого набора.

### 10.5. Нет periodic full reconciliation

Сейчас система умеет:

- ретраить уже существующие pending-задачи;
- перестраивать их после конкретных бизнес-событий.

Но она не делает отдельный регулярный full scan всех подписок, чтобы восстанавливать вообще отсутствующие строки `subscription_nodes`, если они были потеряны или не созданы.

Риск:

- полностью отсутствующая задача не будет автоматически восстановлена, пока не произойдет новое бизнес-событие или ручное вмешательство.

Почему пока оставлено:

- это уже отдельный reconciliation job, а не локальная правка текущего sync-пайплайна.

## 11. Операционные выводы

Что важно понимать при эксплуатации:

- синхронизация нод не является транзакционной частью бизнес-операции;
- успешная активация/смена тарифа не означает мгновенную консистентность на всех нодах;
- pending-записи и retry-метаданные — ключевой источник правды для разбора проблем;
- отсутствие terminal-state означает, что зависшие ошибки надо отслеживать по backlog, `retry_count` и `last_error`.

## 12. Что проверено тестами

Покрыты сценарии:

- `pending_add` success;
- `pending_remove` success;
- `already exists -> fallback update -> success`;
- `already exists -> fallback update -> fail`;
- `not found` на remove как идемпотентный успех;
- отсутствие runtime VPN client для `pending_add`, `pending_update`, `pending_remove` с сохранением pending;
- aggregation ошибок через `errors.Join`;
- sentinel error classification в `vpn`.

## 13. Краткое резюме

Текущий sync-процесс стал заметно безопаснее, чем был изначально:

- убран ложный `active`;
- убрано тихое удаление pending-задач;
- сервис больше не зависит напрямую от текстов внешних ошибок;
- есть защита от двойной обработки внутри процесса;
- частичные ошибки больше не теряются полностью.

При этом система все еще остается intentionally простой:

- без terminal-state;
- без distributed locking;
- без periodic full reconciliation;
- с локализованной, но все еще текстовой классификацией внешних VPN-ошибок.

Это нужно учитывать как осознанный технический долг, а не как уже решенную часть архитектуры.
