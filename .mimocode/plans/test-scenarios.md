# Тест-план бизнес-сценариев VPN-сервиса

## Назначение

Данный документ предназначен для проверки корректности работы бизнес-логики системы подписок, продуктов, заказов и синхронизации VPN-доступов.

Каждый сценарий должен выполняться на чистой тестовой базе либо в заранее подготовленном состоянии.

---

# 🔴 Сценарий №1. Регистрация нового пользователя

## Подготовка

В системе существует:

* Plan Free
* хотя бы одна Node, доступная для Free

Пользователь отсутствует в БД.

---

## Действия

Выполнить регистрацию нового пользователя через Telegram.

```text
telegram_id = 1001
username = test_user
```

---

## Ожидаемый результат

В таблице subscriptions появилась запись:

```text
telegram_id = 1001
username = test_user
client_id != NULL
plan = Free
```

В таблице subscription_nodes созданы записи для всех нод плана Free.

Статус каждой записи:

```text
pending_add
```

или

```text
active
```

если синхронизация успела выполниться.

---

# 🔴 Сценарий №2. Повторная регистрация существующего пользователя

## Подготовка

Пользователь уже зарегистрирован.

---

## Действия

Повторно вызвать регистрацию.

---

## Ожидаемый результат

Новая подписка не создается.

Количество записей в subscriptions не меняется.

client_id остается прежним.

subscription_nodes не дублируются.

---

# 🔴 Сценарий №3. Получение Premium

## Подготовка

Пользователь находится на плане Free.

В системе существует продукт:

```text

```

---

## Действия

Активировать продукт.

---

## Ожидаемый результат

Создан Order.

Order получает статус:

```text
paid
```

или

```text
activated
```

в зависимости от реализации.

В подписке:

```text
plan = Premium
expires_at > now
```

В subscription_nodes появились записи для новых нод Premium.

Для новых нод:

```text
pending_add
```

или

```text
active
```

# 🔴 Сценарий №4. Продление Premium до окончания срока

## Подготовка

Подписка:

```text

```

---

## Действия

Активировать еще один продукт Premium 30 days.

# 🔴 Сценарий №6. Истечение Premium

## Подготовка

Подписка:

```text
plan = Premium
expires_at < now
```

---

## Действия

Запустить Expire Worker.

---

## Ожидаемый результат

Подписка переводится на Free.

expires_at очищается либо переводится в состояние, предусмотренное архитектурой.

Для Premium-нод создаются операции удаления.

В subscription_nodes появляются:

```text
pending_remove
```
