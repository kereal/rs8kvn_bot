## Задача 1: Единая точка входа — merge Create + GetOrCreateSubscription

**Проблема:** Два метода с пересекающейся ответственностью.

**Решение:**
- `Create()` становится приватным `create()` (DB-first helper)
- `GetOrCreateSubscription()` — публичная точка входа, возвращает `*CreateResult`
- Бот-хендлер (`subscription_handler.go:285`) переключается на `GetOrCreateSubscription()`

**Файлы:**

## Задача 7: SQL-миграция — уникальность и индексы

**Проблема:** Нет UNIQUE на `client_id` и `telegram_id`, нет индекса на `subscription_nodes(status, retry_at)`, нет документации для `plans.traffic_limit`.

**Решение — миграция 023:**

```sql
-- 023_add_unique_constraints_and_indexes.up.sql

-- UNIQUE: каждый VPN-клиент привязан ровно к одной подписке
CREATE UNIQUE INDEX idx_subscriptions_client_id_unique ON subscriptions(client_id);

-- UNIQUE: каждый пользователь Telegram имеет ровно одну подписку
CREATE UNIQUE INDEX idx_subscriptions_telegram_id_unique ON subscriptions(telegram_id);

-- Индекс для sync worker: WHERE status IN (...) AND (retry_at IS NULL OR retry_at <= ?)
CREATE INDEX idx_subscription_nodes_retry ON subscription_nodes(status, retry_at);
```

```sql
-- 023_add_unique_constraints_and_indexes.down.sql
DROP INDEX IF EXISTS idx_subscriptions_client_id_unique;
DROP INDEX IF EXISTS idx_subscriptions_telegram_id_unique;
DROP INDEX IF EXISTS idx_subscription_nodes_retry;
```

**Важно:** Перед применением миграции нужно проверить, нет ли дубликатов в продакшн-БД:
```sql
-- Проверка дубликатов client_id
SELECT client_id, COUNT(*) FROM subscriptions GROUP BY client_id HAVING COUNT(*) > 1;

-- Проверка дубликатов telegram_id
SELECT telegram_id, COUNT(*) FROM subscriptions WHERE telegram_id != 0 GROUP BY telegram_id HAVING COUNT(*) > 1;
```

**Файлы:**
- `internal/database/migrations/023_add_unique_constraints_and_indexes.up.sql` — новый
- `internal/database/migrations/023_add_unique_constraints_and_indexes.down.sql` — новый

## Задача 8: Адаптация кода под UNIQUE telegram_id

**Проблема:** UNIQUE на `telegram_id` конфликтует с текущей логикой: `CreateSubscription` и `BindTrialSubscription` проверяют/ревокают старые подписки. С UNIQUE constraint это избыточно — БД сама не допустит дубликат.

**Решение:**
- `CreateSubscription()` (`internal/database/subscriptions.go:44`): удалить блок `UPDATE status = 'revoked'` — просто CREATE
- `BindTrialSubscription()` (`internal/database/trials.go:125`): удалить блок `UPDATE status = 'revoked'` — просто UPDATE триала
- Обновить GORM-теги в модели `Subscription`:
  - `TelegramID` → `gorm:"uniqueIndex"` (вместо `gorm:"index"`)
  - `ClientID` → `gorm:"size:255;uniqueIndex"` (добавить uniqueIndex)

**Файлы:**
- `internal/database/models.go` — обновить теги
- `internal/database/subscriptions.go` — UPDATE revoked убрать
- `internal/database/trials.go` — UPDATE revoked убрать

## Задача 10: Обновить тесты под UNIQUE telegram_id

**Проблема:** Тесты проверяют `status = 'revoked'` после создания новой подписки. Теперь ревока нет — старая подписка просто перезаписывается UNIQUE constraint (ошибка при дубликате) или удаляется до CREATE.

**Подход:**
- Тесты, проверяющие `revokedCount == 1` → проверять что запись удалена или отсутствует
- Тесты, создающие revoked подписку напрямую → другой `telegram_id`
- Тесты, проверяющие `status == "revoked"` после `CreateSubscription` → удалить эти проверки (старой записи больше нет)

- `internal/web/api_test.go`
